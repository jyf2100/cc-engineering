## Q1: Claude Code 中的 Prompt 有哪些类型？

### Prompt 层次结构

```
┌─────────────────────────────────┐
│      System Prompt              │  ← 最顶层：定义 Claude 角色
│  （CLAUDE.md + Skill + Agent）  │
└───────────────┬─────────────────┘
                │
┌───────────────▼─────────────────┐
│      User Prompt                │  ← 用户输入：任务描述
│  （对话中的用户消息）            │
└───────────────┬─────────────────┘
                │
┌───────────────▼─────────────────┐
│      Tool Prompt                │  ← 工具调用：自动生成
│  （工具的 input 参数）           │
└─────────────────────────────────┘
```

### 各层 Prompt 的作用

| 类型 | 来源 | 作用 | 示例 |
|------|------|------|------|
| **System Prompt** | CLAUDE.md/Agent | 定义角色、规范、约束 | "你是代码审查专家" |
| **Skill Prompt** | SKILL.md | 领域知识、工作流程 | "分析财务数据的步骤" |
| **User Prompt** | 用户输入 | 任务描述、问题 | "审查这个函数" |
| **Command Prompt** | commands/*.md | 预定义的命令模板 | /review |

### Prompt 优先级

```
1. CLI --system-prompt（最高）
2. Agent System Prompt
3. SKILL.md 内容
4. CLAUDE.md 内容
5. 默认 System Prompt（最低）
```

## Q2: 如何设计有效的 System Prompt？

### System Prompt 结构

```markdown
# 角色定义
你是一个专业的代码审查专家，专注于发现安全和性能问题。

## 职责范围
- 审查代码质量
- 发现安全漏洞
- 提出优化建议

## 工作方式
1. 首先理解代码意图
2. 逐行分析潜在问题
3. 提供具体可行的建议

## 输出格式
### 问题列表
| 严重程度 | 问题描述 | 位置 | 修复建议 |
|----------|----------|------|----------|

### 总结
- 问题总数：X
- 高危：Y，中危：Z
```

### 关键要素

| 要素 | 描述 | 示例 |
|------|------|------|
| **角色** | 定义 Claude 的身份 | "你是资深架构师" |
| **职责** | 明确工作范围 | "只负责审查，不负责修改" |
| **约束** | 设置边界 | "不要使用 any 类型" |
| **格式** | 指定输出格式 | "用 Markdown 表格输出" |
| **示例** | 提供参考 | "输出格式如：..." |

### Agent System Prompt 示例

```markdown
# .claude/agents/code-reviewer.md
---
name: code-reviewer
description: 专业的代码审查专家
tools: Read, Grep, Glob
model: sonnet
---

## 角色
你是一个有 10 年经验的代码审查专家。

## 审查维度
1. **安全性**：SQL 注入、XSS、敏感信息泄露
2. **性能**：N+1 查询、内存泄漏、循环优化
3. **可维护性**：命名、注释、代码结构
4. **最佳实践**：设计模式、SOLID 原则

## 输出规范
- 使用中文
- 每个问题包含：位置、描述、严重程度、建议
- 最后给出整体评分（1-10）

## 禁止
- 不修改代码
- 不执行命令
- 不做无关评论
```

## Q3: 如何使用 Few-shot 示例？

### Few-shot 原理

```
通过提供示例，让 Claude 理解期望的输出格式和风格。

示例输入 → 示例输出
示例输入 → 示例输出
实际输入 → ?（Claude 会模仿格式）
```

### 在 CLAUDE.md 中使用

```markdown
# CLAUDE.md

## 代码注释规范

### 示例 1：函数注释
```typescript
/**
 * 计算订单总价
 * @param items - 订单项列表
 * @returns 总价（含税）
 * @example
 * calculateTotal([{ price: 100, quantity: 2 }])
 * // 返回 220（含 10% 税）
 */
function calculateTotal(items: OrderItem[]): number {
  // ...
}
```

### 示例 2：类注释
（更多示例...）

写代码时请遵循以上注释风格。
```

### 在 Skill 中使用

```markdown
# .claude/skills/api-doc/SKILL.md
---
description: API 文档生成
---

## 文档格式示例

### 输入
```typescript
function getUser(id: string): Promise<User>
```

### 输出
```markdown
## GET /users/:id

获取指定用户信息

### 参数
| 名称 | 类型 | 必填 | 描述 |
|------|------|------|------|
| id | string | 是 | 用户 ID |

### 返回
```json
{
  "id": "123",
  "name": "张三",
  "email": "zhangsan@example.com"
}
```

### 错误码
- 404: 用户不存在
- 500: 服务器错误
```
```

### Command 中的 Few-shot

```markdown
# .claude/commands/commit.md
---
description: 生成符合规范的 commit message
---

根据代码变更生成 commit message。

## 格式要求
```
<type>(<scope>): <subject>

<body>

<footer>
```

## 示例
输入：修改了用户登录逻辑
输出：
```
feat(auth): 优化用户登录体验

- 添加记住我功能
- 优化错误提示
- 增加登录日志

Closes #123
```
```

## Q4: 如何使用 Chain-of-Thought？

### CoT 原理

```
让 Claude "思考 aloud"，展示推理过程：

问题 → 思考步骤 → 中间结论 → 最终答案
```

### 引导 CoT

```markdown
# 在 System Prompt 中

## 思考流程
在回答之前，请：
1. 理解问题的核心
2. 列出可能的方案
3. 分析每个方案的优缺点
4. 选择最佳方案
5. 解释选择原因

## 输出格式
### 思考过程
1. ...
2. ...

### 结论
...
```

### 复杂任务的 CoT

```
用户：这个性能问题怎么解决？

Claude：
### 分析过程
1. 首先看代码结构...
   - 发现 N+1 查询模式
   - 循环中有数据库调用

2. 评估影响...
   - 每次循环 100ms
   - 总共 1000 次循环
   - 预估总时间 100 秒

3. 考虑解决方案...
   - 方案 A：批量查询（推荐）
     优点：减少到 1 次查询
     缺点：需要重构代码
   - 方案 B：缓存
     优点：实现简单
     缺点：数据可能过期

### 推荐方案
使用方案 A（批量查询），因为...
```

### Agent 中的 CoT

```markdown
# .claude/agents/debugger.md
---
name: debugger
description: 调试复杂问题
---

## 调试流程
当分析 bug 时，请按以下步骤思考：

1. **复现问题**
   - 问题的症状是什么？
   - 什么条件下出现？

2. **定位原因**
   - 可能的原因有哪些？
   - 如何排除不相关的？

3. **验证假设**
   - 如何验证每个假设？
   - 需要检查哪些代码？

4. **制定方案**
   - 修复方案是什么？
   - 有什么副作用？

每步都要说明你的思考过程。
```

## Q5: 如何处理模糊需求？

### 澄清策略

```markdown
# System Prompt 中的澄清规则

## 遇到模糊需求时
在开始工作之前，先确认：
1. 具体范围是什么？
2. 有什么约束条件？
3. 期望的输出格式？
4. 有参考示例吗？

不要猜测，先问清楚。
```

### 澄清示例

```
用户：优化这个函数

Claude：好的，为了更好地优化，我需要确认几个问题：

1. **优化目标**：是优化速度、内存还是可读性？
2. **约束条件**：是否需要保持接口兼容？
3. **数据规模**：通常处理多少数据？
4. **性能要求**：有具体的目标指标吗？

请告诉我更多细节，我会提供更有针对性的优化方案。
```

### 主动引导

```markdown
# CLAUDE.md

## 需求澄清
当用户给出模糊任务时，主动引导：

### 模糊：审查代码
引导：请问是审查安全性、性能还是代码风格？

### 模糊：优化性能
引导：当前的性能瓶颈是什么？有什么指标？

### 模糊：重构
引导：重构的目标是什么？减少耦合、提高可测试性还是其他？
```

## Q6: 如何控制输出格式？

### 格式约束

```markdown
# 输出格式规范

## 代码审查报告
### 问题列表
| 严重程度 | 文件 | 行号 | 问题描述 | 建议 |
|----------|------|------|----------|------|
| 高 | auth.ts | 45 | 硬编码密钥 | 使用环境变量 |

### 总结
- 严重程度分布：高 X，中 Y，低 Z
- 整体评分：N/10
- 优先修复：[问题 ID 列表]
```

### JSON 输出

```bash
# 强制 JSON 输出
claude -p "分析代码并输出 JSON" \
  --output-format json \
  --json-schema '{
    "type": "object",
    "properties": {
      "issues": {"type": "array"},
      "score": {"type": "number"}
    }
  }'
```

### 结构化输出模板

```markdown
# .claude/commands/analyze.md
---
description: 代码分析报告
---

分析代码并输出结构化报告。

## 输出模板
```markdown
# 分析报告

## 概览
- 文件数：N
- 代码行数：M
- 语言：X, Y, Z

## 问题
### 高优先级
1. [问题描述] (位置)
2. ...

### 中优先级
...

## 建议
1. ...
2. ...

## 评分
- 代码质量：X/10
- 可维护性：Y/10
- 安全性：Z/10
```
```

## Q7: 如何处理多轮对话？

### 上下文管理

```
用户：分析 src/auth.ts
Claude：（分析完成）

用户：有什么安全问题？
Claude：（基于之前的分析回答）

用户：怎么修复？
Claude：（提供具体修复方案）
```

### 引用之前的内容

```
# 正确：引用上下文
用户：修复上面提到的第一个问题
Claude：好的，我来修复第一个问题（硬编码密钥）...

# 错误：重复描述
用户：修复 src/auth.ts 第 45 行的硬编码密钥问题
（浪费 Token 重复已知信息）
```

### 会话续接

```bash
# 继续上次会话
claude -c -p "继续分析其他模块"

# 恢复指定会话
claude -r abc123 -p "基于上次的分析，提出改进建议"
```

### 避免上下文丢失

```markdown
# CLAUDE.md

## 长任务处理
当任务较长时：
1. 分阶段执行
2. 每阶段输出中间结论
3. 下阶段引用中间结论
4. 避免重复分析

## 会话管理
- 重要结论记录到文件
- 长任务拆分成多个会话
- 使用 /continue 恢复上下文
```
